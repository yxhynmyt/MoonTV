# Code Reuse 思考指南

> **目的**：在写新代码前先停一下，确认这段东西是不是已经存在。

---

## 问题本质

**重复代码是最常见的一致性 bug 来源。**

当你复制粘贴已有逻辑，或者把已有逻辑换个地方重写一遍时：

- bug fix 无法同步传播
- 不同实现会逐渐偏离
- 代码库会越来越难理解

---

## 写新代码前

### 第一步：先搜索

```bash
# 搜类似函数名
rg "functionName" .

# 搜类似逻辑关键词
rg "keyword" .
```

### 第二步：先问自己这些问题

| 问题                               | 如果答案是 Yes               |
| ---------------------------------- | ---------------------------- |
| 是否已经存在相似函数？             | 直接复用或在原实现上扩展     |
| 这种模式是否在别处已经出现？       | 跟随现有模式，不要自创新写法 |
| 它是否应该成为 shared utility？    | 放到正确的共享位置           |
| 我是不是正在复制另一个文件的代码？ | **立刻停下**，改成抽共享     |

---

## 常见重复模式

### 模式 1：复制函数

**Bad**：把校验函数复制到另一个文件

**Good**：抽到 shared utility，再按需 import

### 模式 2：高度相似的组件

**Bad**：新建一个和现有组件 80% 相似的新组件

**Good**：通过 props / variant 扩展已有组件

### 模式 3：重复常量

**Bad**：在多个文件里各自定义同一个常量

**Good**：只保留一个 single source of truth，其余位置统一 import

---

## 什么时候应该抽象

**适合抽象的情况**：

- 同一段代码已经出现 3 次以上
- 逻辑足够复杂，已经值得担心 bug
- 很可能有多人、多处会继续用到

**不适合抽象的情况**：

- 只用一次
- 只是简单一行
- 抽象后的复杂度反而高于重复本身

---

## 批量修改之后

如果你刚对多个文件做了相似修改：

1. **Review**：确认是不是所有实例都改到了
2. **Search**：再跑一次 `rg` 查漏
3. **Consider**：这是不是已经到了该抽象的时候

---

## 易错点：不同机制产出同一结果

**问题**：当两套不同机制需要产出同一批文件时，例如初始化用递归目录复制，而更新流程用手写 `files.set()`，那么目录结构变化只会自动传播到“自动机制”那一边，手写那一边会静默漂移。

**表现**：init 完全正常，但 update 会把文件生成到错误路径，或者漏掉部分文件。

**预防清单**：

- [ ] 迁移目录结构时，搜索所有引用旧结构的代码路径
- [ ] 如果一边是自动推导（glob / copy），另一边是手写列表，那么手写那边必须同步更新
- [ ] 增加一个 regression test，对比两套机制的产出是否一致

---

## 提交前检查

- [ ] 已搜索过现有相似实现
- [ ] 没有应该抽共享却仍在复制粘贴的逻辑
- [ ] 常量只定义在一个地方
- [ ] 相似模式遵循同一种结构
