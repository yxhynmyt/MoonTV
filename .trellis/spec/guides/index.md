# 思考指南

> **目的**：在动手前，把那些最容易“没想到”的风险提前想出来。

---

## 为什么需要思考指南？

大多数 bug 和技术债，并不是“不会写”，而是“当时没想到”：

- 没想到层与层之间的边界问题，就会出现 cross-layer bug
- 没想到代码模式正在重复，就会出现到处复制的实现
- 没想到边界条件，就会在运行时出错
- 没想到未来维护者，就会留下难读又难改的代码

这些指南的作用，是让你在编码前先问对问题。

## 快速判断：什么时候该看这些指南？

### 什么时候要考虑 cross-layer 问题

- [ ] Feature 同时触及 3 层以上，例如 API、Service、Component、Database
- [ ] 数据格式在多个层之间发生变化
- [ ] 多个消费者需要同一份数据
- [ ] 某段逻辑应该放哪一层你已经拿不准了

这时请读 [Cross-Layer 思考指南](./cross-layer-thinking-guide.md)。

### 什么时候要考虑代码复用

- [ ] 你正在写的逻辑和仓库里已有实现很像
- [ ] 同一种模式已经重复 3 次以上
- [ ] 你正在给多个位置同时加新字段
- [ ] **你正在修改某个 constant 或 config**
- [ ] **你准备新建 utility / helper function**，但还没先搜仓库

这时请读 [Code Reuse 思考指南](./code-reuse-thinking-guide.md)。

## 修改前规则（关键）

> **改任何值之前，先全局搜索。**

```bash
# 先搜索你准备修改的值
rg "value_to_change" .
```

这个习惯能避免绝大多数“漏改某个引用点”的 bug。

## 如何使用这个目录

1. **编码前**：快速浏览相关 thinking guide
2. **编码中**：一旦感觉逻辑在重复、边界在变复杂，就回来看指南
3. **出过 bug 之后**：把这次教训沉淀回对应指南
