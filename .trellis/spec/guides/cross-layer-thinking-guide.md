# Cross-Layer 思考指南

> **目的**：在实现前先把跨层数据流和边界责任想清楚。

---

## 问题本质

**大多数 bug 出现在层与层的边界，而不是层内部。**

常见的 cross-layer bug 包括：

- API 返回的是格式 A，frontend 却按格式 B 读取
- Database 存的是 X，service 转成 Y 时把信息丢了
- 多个层各自实现了“看起来一样”的逻辑，但行为并不一致

---

## 实现 cross-layer 功能前

### 第一步：先画出数据流

把数据是怎么流动的写出来：

```text
Source -> Transform -> Store -> Retrieve -> Transform -> Display
```

对每一个箭头都问自己：

- 数据此时是什么格式？
- 这里可能出什么错？
- 这一段的校验责任归谁？

### 第二步：找出边界

| Boundary               | 常见问题             |
| ---------------------- | -------------------- |
| API -> Service         | 类型不匹配、字段缺失 |
| Service -> Database    | 格式转换、null 处理  |
| Backend -> Frontend    | 序列化、日期格式     |
| Component -> Component | props shape 变化     |

### 第三步：定义契约

对每个边界都明确下来：

- 精确输入格式是什么？
- 精确输出格式是什么？
- 这里可能出现哪些错误？

---

## 常见 cross-layer 错误

### 错误 1：默认假设格式一定正确

**Bad**：不检查就默认日期格式一定符合预期

**Good**：在边界处做显式格式转换

### 错误 2：校验散落在多个层

**Bad**：同一件事在多个层都重复校验

**Good**：在入口边界校验一次，并明确责任

### 错误 3：抽象泄漏

**Bad**：Component 直接知道 database schema

**Good**：每一层只了解它相邻层的契约

## Cross-Layer 功能检查清单

实现前：

- [ ] 已画出完整数据流
- [ ] 已识别所有层间边界
- [ ] 已定义每个边界的格式
- [ ] 已明确校验发生在哪一层

实现后：

- [ ] 已用边界条件测试：null、empty、invalid
- [ ] 已验证每个边界的错误处理
- [ ] 已确认数据 round-trip 后不会丢失

---

## 什么时候要补 Flow 文档

以下情况应补更详细的 flow 文档：

- Feature 横跨 3 层以上
- 有多个团队或多人协作
- 数据格式本身很复杂
- 这个功能过去已经反复出过 bug
